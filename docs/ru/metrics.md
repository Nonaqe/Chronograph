# Метрики

[← Оглавление](README.md) · [English](../en/metrics.md)

У каждого сигнала Chronograph есть **прозрачное задокументированное определение**. Никакого непрозрачного «health score». Здесь — точная формула и обоснование (*почему* метрика коррелирует с реальной проблемой) для каждого сигнала.

Сквозное правило: **все временные окна отсчитываются от `anchor = max(committed_at)`** (последняя активность репозитория), а не от wall‑clock «сейчас». Это гарантирует, что один и тот же репозиторий всегда даёт одни и те же числа ([детерминизм](architecture.md#детерминизм)).

## Churn

**Что измеряет:** как часто и насколько сильно меняется файл.

**Определение:** для каждого файла churn — это **число коммитов, тронувших его**, посчитанное по всей истории и по трём скользящим окнам:

| Поле | Окно |
|---|---|
| `churn_total` | вся история |
| `churn_30d` | последние 30 дней до якоря |
| `churn_90d` | последние 90 дней |
| `churn_365d` | последние 365 дней |

Окна конфигурируемы (дефолты 30/90/365 из ТЗ). Chronograph также считает суммарные добавленные/удалённые строки.

**Непрерывность при переименованиях:** churn следует за переименованиями. Когда `b.txt` переименован в `c.txt`, их истории склеиваются в один канонический файл, а не фрагментируются на два — иначе переименованный файл выглядел бы искусственно «молодым» и малочёрновым. Ради этого переименования сохраняются (`old_path`) при ingestion.

**Почему коррелирует с проблемами:** часто меняющийся код — там, где концентрируются баги и где цена непонимания платится снова и снова. Один churn ещё не риск (постоянно трогаемый, но тривиальный файл — нормально) — риском он становится вместе со сложностью ([hotspots](#hotspots)).

По умолчанию [механические коммиты](#механические-коммиты) исключаются из churn.

## Complexity

**Что измеряет:** структурную запутанность текущего содержимого файла.

**Определение — цикломатическая сложность** (для поддержанных языков): `1 + число узлов‑ветвлений` в AST, где узлы‑ветвления — это `if`/`elif`, циклы, ветви `match`/`switch`/`select`, `except`/`catch` и тернарный оператор. AST строится **tree‑sitter**, а не регэкспами.

Поддержанные для цикломатической сложности языки:

- **Rust**, **Python**, **Go**, **JavaScript/TypeScript** (вкл. TSX).

**Fallback:** для прочих языков или при ошибке парсинга Chronograph использует **эвристику по отступам** (сумма глубин вложенности по непустым строкам, через относительный стек отступов — устойчиво к табам/пробелам). Грубый, но ненулевой сигнал.

`complexity_per_loc` нормализует сложность на число строк кода.

**Источник контента:** сложность считается из **git‑блоба** текущего состояния файла (адресуется `blob_sha`), а не из рабочей директории. Это делает её детерминированной и исторически точной и позволяет кэшировать результат по `(blob_sha, язык)` — один и тот же контент не парсится дважды.

**Известные ограничения (честно задокументированы):**

- Логические `&&`/`||` в v1 **не** учитываются (грамматики не выделяют их отдельными узлами).
- Тела макросов (например, Rust `macro_rules!`) — это token trees, не control flow, поэтому ветвления внутри макросов не считаются.

## Hotspots

**Что измеряет:** пересечение churn и complexity — код, который постоянно меняют и меньше всего понимают.

**Формула:**

```
score = перцентиль_churn × перцентиль_complexity
```

где каждый перцентиль — **ранг‑перцентиль в [0, 1]**: позиция файла в отсортированном порядке, делённая на `N − 1` (тай‑брейк по пути для детерминизма). Поэтому score тоже в `[0, 1]` и является ровно произведением двух раскрываемых составляющих — всегда видно, *почему* файл на своём месте (его churn, complexity и каждый перцентиль).

**Ранжируются только:** живые файлы с **цикломатической** сложностью (четыре поддержанных языка). Файлы на indentation‑fallback (доки, конфиги, снапшоты, неподдержанные языки) из hotspot‑рейтинга **исключены**.

> **Почему исключаем fallback‑файлы?** Эвристика по отступам и цикломатическая сложность — разные шкалы. На полном прогоне большой YAML или LICENSE может набрать огромную «сложность» по отступам и, вместе с релизным churn‑шумом, дать ложные hotspot'ы. Исключение fallback‑файлов сохраняет осмысленность рейтинга (реальный код). Будущее улучшение — *allowlist* расширений языков программирования, чтобы реальный код на пока неподдержанных языках (C++, Ruby, Java…) тоже ранжировался, а не‑код оставался вне.

**Никакого магического числа:** score никогда не масштабируется в 0–100. Это всегда видимое произведение двух видимых перцентилей.

## Change coupling

**Что измеряет:** какие файлы склонны меняться **в одном коммите** — прокси скрытой логической/архитектурной зависимости.

**Алгоритм:** co‑occurrence по коммитам (self‑join таблицы изменений по SHA коммита, берём пары с `a.path < b.path`). Рассматриваются только реально со‑встречавшиеся пары — сложность `O(Σ kᵢ²)` по размерам коммитов, а **не** `O(файлов²)`.

**Формулы:**

```
support(A, B)       = число коммитов, тронувших и A, и B
coupling_ratio(A,B) = support(A, B) / min(commits(A), commits(B))
```

Поскольку `support ≤ min(commits)`, ratio всегда в `(0, 1]`. Ratio около 1.0 означает «почти каждый раз, когда менялся один, менялся и другой».

**Симметрия:** `coupling(A, B) == coupling(B, A)` по построению (пара канонична `a < b`; support — симметричное пересечение; `min` симметричен). Инвариант закреплён property‑тестом.

**Пороги:** паре нужен `support ≥ min_support` (по умолчанию **5**) — это отсекает случайные разовые со‑изменения. Механические/гигантские коммиты исключаются по умолчанию (они и искажают сигнал, и раздувают стоимость `O(kᵢ²)`).

**Не только код:** в отличие от hotspots, coupling осмыслен и для конфигов и тестовых снапшотов (например, пара `*.rs` ↔ `*.stderr` UI‑тестов — легитимна).

**Почему коррелирует с проблемами:** два файла, которые всегда меняются вместе, но лежат в разных модулях, сигналят о зависимости, которую архитектура не делает явной — той самой сцепленности, из‑за которой изменения расходятся неожиданно.

## Knowledge / bus factor

**Что измеряет:** насколько концентрированы знания о файле — *риск* того, что они живут в слишком малом числе голов.

**Определение:** из `git blame` Chronograph считает **владение** каждого автора (долю живых строк) по файлу, затем:

```
bus_factor        = минимальное число топ-владельцев, чья суммарная доля
                    СТРОГО превышает порог (дефолт 0.5, т.е. «> 50% знаний»)
top_owner_ratio   = доля единственного крупнейшего владельца
```

**Bus factor = 1** означает, что один человек владеет более чем половиной файла — классический риск концентрации.

**Подаётся как риск, никогда как заслуга или вина.** Вывод сортируется по риску (сначала минимальный bus factor, затем максимальная доля топ‑владельца). **Авторы анонимизированы по умолчанию** (`Author #N`); реальные имена — только по явному флагу `--show-names`. Это продуктовое требование — показывать «кто написал больше всех» подорвало бы adoption и не в этом смысл.

**Бюджет blame:** blame патологически большого, часто переписываемого файла (например, многомегабайтного генерируемого `CHANGELOG` с сотнями ревизий) неделим и может занимать десятки минут. Такие файлы пропускаются, когда их стоимость (`ревизии × добавленные строки`) превышает [бюджет blame](configuration.md#бюджет-blame) (по умолчанию 10 000 000), и указываются явно — никогда не выбрасываются молча.

## Code age

**Что измеряет:** возраст живых строк файла — сигнал стабильности, дополняющий churn.

**Определение:** из `git blame` каждая живая строка получает возраст в **днях от якоря** (`max(committed_at)`). По файлу Chronograph отдаёт распределение перцентилями:

| Поле | Значение |
|---|---|
| `newest_age_days` | самая молодая живая строка |
| `median_age_days` | медианный возраст строк |
| `p90_age_days` | возраст 90‑го перцентиля |
| `oldest_age_days` | самая старая живая строка |

Перцентили описывают распределение без произвольного порога «% старше X дней».

**Интерпретация:**

- **Малый медианный возраст** → зона постоянного переписывания (часто пересекается с hotspots).
- **Большой медианный возраст** → стабильный, зрелый, устоявшийся код.

## Механические коммиты

Некоторые коммиты «механические»: массовые переименования, прогоны форматирования, дропы vendored‑зависимостей или релизные бампы, трогающие сразу огромное число файлов. Оставленные как есть, они искажают churn и coupling.

Chronograph помечает коммит механическим, когда он трогает больше **N** файлов, где **N** задаётся через `--mechanical-max-files`. **По умолчанию эвристика выключена** (порога нет), потому что хороший дефолт зависит от репозитория, а ТЗ его не фиксирует — Chronograph не выдумывает магическое число. При включении churn и coupling по умолчанию исключают механические коммиты.

См. [Конфигурацию](configuration.md#механические-коммиты).
