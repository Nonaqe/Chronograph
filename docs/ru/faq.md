# FAQ и решение проблем

[← Оглавление](README.md) · [English](../en/faq.md)

## Общее

### Chronograph оценивает разработчиков?

Нет — и намеренно не может. Метрики про **файлы и модули**, не про людей. Единственный близкий к людям сигнал, карта знаний, подаётся только как **риск концентрации** (bus factor), с авторами **анонимными по умолчанию** (`Author #N`). Это продуктовое требование, а не вкусовщина.

### Какие языки поддержаны?

Для **цикломатической сложности** (и, соответственно, hotspot‑рейтинга): **Rust, Python, Go, JavaScript/TypeScript** (вкл. TSX). Прочие языки получают fallback‑сложность по отступам, но исключены из hotspot‑рейтинга. Churn, coupling, knowledge и code age работают для любого файла.

### Он вызывает бинарь `git`?

Нет. Chronograph читает репозитории напрямую через **gix (gitoxide)**. CLI `git` используется только для сборки тестовых фикстур.

### Мои данные куда‑то отправляются?

Нет. Всё работает локально. HTML‑отчёт и JSON‑экспорт — самодостаточные файлы с нулём внешних запросов. (GitHub Action скачивает *бинарь* из публичного релиза; ваш код он никогда не выгружает.)

## Детерминизм и воспроизводимость

### Почему два прогона дают идентичный вывод?

По замыслу — один репозиторий + конфиг → **байт‑идентичные** результаты. Таймстемпы в UTC, порядок обхода фиксирован, агрегации тай‑брейкаются детерминированно, сериализация стабильна. См. [Архитектура → Детерминизм](architecture.md#детерминизм).

### Анимация в web UI каждый раз выглядит иначе — это баг?

Нет. Байт‑идентичность относится к **артефакту `chronograph.json`**, а не к живому UI. Рендер роста/timeline использует физику и время и намеренно не воспроизводим пиксель‑в‑пиксель.

## Решение проблем

### «not a git repository» / ошибки открытия

Наведите Chronograph на корень репозитория (директорию с `.git`) и убедитесь, что это настоящий git‑репозиторий. В CI не забудьте `fetch-depth: 0`, чтобы присутствовала полная история.

### GitHub Action не может скачать бинарь (404)

Action скачивает релизный ассет анонимным `curl`, поэтому **upstream‑репозиторий обязан быть публичным**, и для запинённой `version`/тега должен существовать соответствующий релиз. На не‑Linux‑x64 runner'ах Action намеренно падает с понятным сообщением.

### Первая сборка идёт вечно

Chronograph при первой сборке компилирует **встроенный DuckDB** из исходников (плюс грамматики tree‑sitter и gix). Это одноразовая цена; последующие сборки инкрементальны. Избегайте `cargo clean`, который выбрасывает этот кэш.

### `cargo test --workspace` иногда падает на Windows с ошибкой rlib

Это известная гонка сборки cargo, когда много тяжёлых тест‑бинарей линкуются параллельно (каждый статически линкует duckdb/arrow/gix). Это **не** ошибка кода. Обходные пути: сборка последовательно (`cargo test -j 1`) или прогон тестов по одному крейту. Обычно сходится за один ретрай.

### Некоторых файлов нет в таблицах knowledge / age

Скорее всего, они **пропущены бюджетом blame** (очень большие, часто переписываемые файлы). Проверьте сводку пропусков / список `blame_skips` и повысьте или отключите бюджет через `--blame-budget` (см. [Конфигурацию](configuration.md#бюджет-blame)).

### Ошибки схемы / кэша после апгрейда

Удалите кэш (`<repo>/.chronograph/`) и перезапустите, или используйте `--no-incremental` для форсированного полного пересчёта.

## Анти‑цели — чем Chronograph не станет

- **Индивидуальная оценка продуктивности разработчиков.**
- **Менеджерские DORA‑метрики** (это рынок LinearB/Swarmia).
- **Линтер реального времени** (это SonarQube).
- **«Все языки сразу»** — стартовый набор: JS/TS, Python, Go, Rust.
- **ML‑предсказание дефектов** (отдельная задача, нужен датасет багов).

## Советы по производительности

- Всегда используйте **release**‑бинарь; debug в 5–20 раз медленнее на CPU‑bound путях.
- Используйте `--exclude`, чтобы убрать vendored/generated деревья.
- Включайте `--mechanical-max-files` на репозиториях с массовыми коммитами.
- Настраивайте `--blame-budget`, если knowledge/age медленно идут на гигантских файлах.
- Повторные прогоны инкрементальны — обрабатываются только новые коммиты.
